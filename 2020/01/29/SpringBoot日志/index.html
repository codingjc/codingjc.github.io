SpringBoot日志

1、日志框架

市面上的日志框架：

JUL、JCL、Jboss-logging、logback、log4j、log4j2、slf4j…

日志门面（日志的抽象层） 日志实现
JCL(Jakata Commons Logging) 、SLF4j(Simple Logging Facade for Java)、Jboss-logging Log4j、JUL(java.util.logging)、log4j2、Logback

左边选一个门面(抽象层)、右边来选一个实现；

日志门面：SLF4J

日志实现：Logback

SpringBoot：底层是Spring框架，Spring框架默认是用JCL;

SpringBoot选用的是SLF4J和Logback

2、SLF4J使用

1、如何在系统中使用SLF4J

在开发的时候，日志记录方法的调用，不应该来直接调用日志的实现类，而是调用日志抽象层里面的方法；

导入slf4j的jar和logback的实现jar

SLF4J关系图

Logback：应用程序直接调用SLF4J抽象层的方法，实现的是Logback。导入slf4j和logback包

Log4j：应用程序直接调用SLF4J抽象层的方法，实现的是log4j，由于slf4j后发行，所以如果使用log4j，则需要加入slf4j-log4j的适配包。导入slf4j和slf4j-log412和log4j包

java.util.logging：应用程序直接调用SLF4j抽象层的方法，实现的是java.util.logging。需要导入slf4j和slf4j-jdk14包

说明：每一个日志的实现框架都有自己的配置文件，使用slf4j以后，配置文件还是做成日志实现框架的配置文件 (最终还是使用实现框架的配置文件)

2、遗留问题

场景：开发某系统采用的是slf4j+logback：Spring（commons-logging）、Hibernate（jBoss-logging）、Mybatis、xxx

统一日志记录，别的框架和我一起统一使用slf4j进行输出

思路：先把Spring框架中的Commons logging包先排除掉，再用jcl-over-slf4j包来替换

1、将系统中其他日志框架先排除出去；

2、用中间包来替代原有的日志框架；

3、导入slf4j的实现框架

3、SpringBoot日志分析

每一个starter中都依赖此starter

在spring-boot-starter中又依赖了spring-boot-starter-logging

总结：

1、Springboot底层也采用slf4j+logback

2、SpringBoot也将其他日志都替换为slf4j

4、SpringBoot默认配置

1、日志的级别（trace < debug < info < warn < error）

2、默认的日志级别是info

3、如果logging.file.name和logging.path都没指定的话，旨在控制台输出；file.name和path不能同时设置，一般设置path，同时设置的话，file起作用。file.name=E:/log/sprinboot.log

logging.file.name logging.file.path example description
none none 控制台输出
指定文件名 none my.log 输出日志到文件，在当前项目下
none 指定路径 /spring/log 输出到当前磁盘根目录下的spring.log文件

SpringBoot修改日志的默认配置

5、自定义配置文件

在类路径下(也就是在resources下)放上每个日志框架自己的配置文件即可；SpringBoot就不再使用默认的配置了

Logging System Customization
Logback logback-spring.xml, logback-spring.groovy, logback.xml, or logback.groovy
Log4j2 log4j2-spring.xml or log4j2.xml
JDK (Java Util Logging) logging.properties

eg：logback.xml

新特性：官方推荐使用logback-spring.xml

logback.xml：直接就被日志框架识别了

logback-spring.xml：日志框架不直接加载日志的配置项，由SpringBoot解析日志配置，可以使用SpringBoot的高级功能
